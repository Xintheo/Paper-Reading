# 任务

基于下方提供的多篇论文的「数据可用性」相关段落（Data availability、Code availability、Supplementary material 等），为每篇论文生成一张「可用数据表」。

# 输出格式

每篇论文输出一张表，标题格式：

## 文献N (PMID: xxxxxxxx) 可用数据表

| 数据类型 | 名称编号 | 链接 | 说明 |
|---|---|---|---|

**markdown 表格格式硬性要求**：
- 每行独占一个文本行，行之间用真正的换行符分隔，绝不允许把多行用 `|` 拼到同一行
- 行内单元格之间用 ` | ` 分隔，行首和行尾各有一个 `|`
- 单元格内不能出现裸 `|` 字符（如有，用 `\|` 转义）
- 单元格内不能出现换行（如有，用空格替代）

# 列填写规则

## 数据类型 列（必须从下方枚举中选一个）

数据库类：
- `GEO` — NCBI Gene Expression Omnibus（GSE/GSM/GPL 编号）
- `SRA` — NCBI Sequence Read Archive（SRR/SRX/SRP/PRJNA 编号）
- `dbGaP` — NCBI 表型基因型（phs 编号）
- `ArrayExpress` — EBI 微阵列（E-MTAB 等）
- `ENA` — European Nucleotide Archive
- `PRIDE` — 蛋白质组数据（PXD 编号）
- `MetaboLights` — 代谢组数据（MTBLS 编号）
- `UK Biobank` / `TCGA` / `GTEx` / `1000 Genomes` — 大型公开队列/项目
- `ClinVar` / `dbSNP` / `gnomAD` — 变异数据库

仓库类：
- `Zenodo`
- `Figshare`
- `Dryad`
- `OSF` — Open Science Framework

代码类：
- `GitHub`
- `GitLab`
- `Bitbucket`

其他：
- `Supplementary` — 随文补充材料（包括 Supplementary Tables/Figures/Data files）
- `Upon request` — 仅"按需索取"，无公开访问点
- `Web tool` — 在线工具/网站（非数据库）
- `其他` — 不属于以上任何一类时使用，并在「说明」列注明实际类型

## 名称编号 列

- 填具体的访问号或仓库名，如 `GSE12345`、`PRJNA999`、`PXD012345`、`example/repo`（GitHub 仓库名）
- 如果是 Supplementary，填具体文件名或编号，如 `Supplementary Table S1`、`Supplementary Data 1`
- 如果是 Upon request，留空
- 不要带链接 markdown，链接放下一列

## 链接 列

- 完整 URL，保留原文中的链接（如果原文有）
- 如果原文只给了访问号没给链接，**根据数据库标准 URL 格式补全**：
  - GEO: `https://www.ncbi.nlm.nih.gov/geo/query/acc.cgi?acc=GSE12345`
  - SRA/PRJNA: `https://www.ncbi.nlm.nih.gov/bioproject/PRJNA999`
  - PRIDE: `https://www.ebi.ac.uk/pride/archive/projects/PXD012345`
  - GitHub: `https://github.com/example/repo`
- 如果是 Supplementary 且原文给了 DOI/URL，填上；否则留空
- 如果是 Upon request，留空
- 链接用纯 URL 格式，不要用 markdown 链接语法 `[...](...)`

## 说明 列

简短描述该数据/资源**是什么**，便于以后回看时一眼识别用途。建议格式：
- 物种 + 样本类型 + 数据类型 + 样本量（如有）
- 例如：`小鼠肝脏 RNA-seq, n=12`、`人类血浆蛋白质组 LC-MS/MS, n=250`、`数据分析与可视化代码`、`小鼠 ALD 模型 scRNA-seq 原始数据`

如果原文没说清楚，能写多少写多少，不要编造。最长 30 字。

# 行的生成规则

## 一篇论文产生几行

- 原文里提到几个独立的数据/代码/资源，就生成几行
- 例：`Data deposited in GEO (GSE12345) and SRA (PRJNA999), code at github.com/abc/def, see Supplementary Table S1` → 4 行
- 例：`Data is available at GEO GSE12345 and code at github.com/abc/def` → 2 行（GEO 1 行 + GitHub 1 行）
- 例：`Data are available from the corresponding author upon reasonable request` → 1 行，数据类型填 `Upon request`，名称/链接留空，说明填 "按需索取"

## 多个编号合并的情况

如果原文是 `GSE12345 and GSE12346`，**拆成 2 行**（每个编号一行）。

如果原文是 `Supplementary Tables S1-S5`，可以**合并成 1 行**，名称编号填 `Supplementary Tables S1-S5`。

# 排序

每张表内部，行按以下顺序排列：
1. 数据库类（GEO、SRA 等）
2. 仓库类（Zenodo、Figshare、Dryad、OSF）
3. 代码类（GitHub、GitLab、Bitbucket）
4. Supplementary
5. Web tool / 其他
6. Upon request（永远排最后）

同一类内部按原文出现顺序。

# 完整示例

输入：
```

## 文献1 (PMID: 41497980)

### Data availability statement

The datasets presented in this study can be found in online repositories. RNA-seq data has been deposited in GEO (accession GSE12345). Single-cell data is available at SRA under PRJNA999888. Analysis code is available at https://github.com/example/ald-analysis. Additional materials can be found in the Supplementary Material.

### Code availability

Custom analysis scripts are deposited at github.com/example/ald-analysis (also referenced above).

```

输出：

## 文献1 (PMID: 41497980) 可用数据表

| 数据类型 | 名称编号 | 链接 | 说明 |
|---|---|---|---|
| GEO | GSE12345 | https://www.ncbi.nlm.nih.gov/geo/query/acc.cgi?acc=GSE12345 | RNA-seq 数据 |
| SRA | PRJNA999888 | https://www.ncbi.nlm.nih.gov/bioproject/PRJNA999888 | 单细胞测序数据 |
| GitHub | example/ald-analysis | https://github.com/example/ald-analysis | 分析代码 |
| Supplementary | Supplementary Material |  | 补充材料 |

注意：
- GitHub 仓库在 Data availability 和 Code availability 里都提到了，**只生成 1 行**，不要重复
- Supplementary 没具体文件名时，名称编号填 `Supplementary Material`，链接留空

# 输出

仅输出表格。每篇论文一张表，表前面带 `## 文献N (PMID: xxxxxxxx) 可用数据表` 标题。不要解释、不要分段说明、不要任何其他内容。
